# Search

Latent Space Hub uses a multi-tier search system combining vector similarity, full-text search, and keyword matching.

## Three Search Tiers

### 1. Vector Search (Semantic)

Finds content by meaning, not just keywords. Uses Turso's native `vector_top_k()` on 1536-dimension embeddings generated by OpenAI's `text-embedding-3-small` model.

- Stored as `F32_BLOB` in the `chunks` table
- Indexed via `libsql_vector_idx`
- Queries return the most semantically similar chunks

### 2. FTS5 (Full-Text Search)

SQLite's built-in full-text search engine. Searches exact words and phrases across chunk text.

- FTS5 virtual table: `chunks_fts` over `chunks.text`
- Fast for keyword-heavy queries
- Supports phrase matching and boolean operators

### 3. Hybrid Search (RRF)

Combines vector and FTS results using **Reciprocal Rank Fusion** — a technique that merges two ranked lists without needing comparable scores.

This is the default search mode. It gives the best results by combining semantic understanding (vector) with keyword precision (FTS).

## Fallback Chain

If a search tier fails or is unavailable, the system degrades gracefully:

```
hybrid (vector + FTS)  →  vector-only  →  FTS-only  →  LIKE (last resort)
```

## What Gets Searched

| Search type | What it searches | Use case |
|-------------|-----------------|----------|
| **Node search** | Titles and descriptions | Finding specific episodes, guests, topics |
| **Content search** | Chunk text (transcripts, articles, papers) | Finding what was said about a topic |

## Via MCP Tools

| Tool | What it does |
|------|-------------|
| `ls_search_nodes` | Keyword search across node titles, descriptions, and notes |
| `ls_search_content` | Full content search through chunks (uses hybrid by default) |
| `ls_sqlite_query` | Custom SQL for advanced filtering and aggregation |

## Via Web UI

- **Cmd+K** opens the global search modal
- Searches across node titles and dimensions
- Results show node title, type badge, and matching dimensions
- Click a result to navigate to the node

## How Embeddings Are Generated

1. Node content is **chunked** (~2000 characters per chunk, 400-character overlap)
2. Each chunk is sent to OpenAI's `text-embedding-3-small` model
3. The 1536-dimension vector is stored as `F32_BLOB` in `chunks.embedding`
4. Turso indexes the vectors via `libsql_vector_idx` for fast retrieval

All ~36,000 chunks in the graph have embeddings. New content is embedded automatically during ingestion via the auto-embed queue.
